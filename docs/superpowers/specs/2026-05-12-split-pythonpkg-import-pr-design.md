# Дизайн: два PR — механический перенос pythonpkg и отдельно доработки

## Контекст

В форк `prawwtocol/otterbrix` нужно разделить работу на два pull request:

1. **PR1** — перенос кода из локального репозитория `pythonpkg` без предметных изменений.
2. **PR2** — все остальные изменения поверх результата PR1 (адаптация к `otterbrix`, сборка, оптимизации, бенчмарки и прочее).

Ориентир снимка `pythonpkg`: **последняя ревизия на диске** (`HEAD` локального клона на момент подготовки PR1). В описании PR1 и в теле коммита указывается **полный hash** этого коммита для воспроизводимости.

## Договорённости по PR1 (механический перенос)

- **Допустимые операции:** добавление и/или перемещение файлов так, чтобы содержимое перенесённых файлов **побайтно/построчно совпадало** с соответствующими файлами в `pythonpkg@<hash>`, с учётом только того, что неизбежно при переносе (например, одна и та же кодировка и переводы строк; без правок «для сборки»).
- **Запрещено в PR1:** любые правки содержимого перенесённых файлов, включая **пути в `#include`**, правки CMake/Conan, правки в коде ядра `otterbrix`, добавление или изменение логики, форматирование, автоматические рефакторинги.
- **Не цель PR1:** успешная сборка или прохождение CI. Ожидаемо, что после чистого переноса проект **может не компилироваться**. В описании PR1 это явно предупредить ревьюера.

## Договорённости по PR2

- Все изменения, необходимые для интеграции в дерево и билд-систему `otterbrix`: инклюды, `CMakeLists.txt`, правки нативной обвязки, изменения в ядре, тесты, бенчмарки, CI, документация по необходимости.
- Базовая ветка для PR2: **вершина ветки PR1** (после merge PR1 в целевую линию разработки или от неё же ответвление).

## Проверка соответствия PR1 снимку

- Для каждого перенесённого пути: сравнение с `pythonpkg@<hash>` (дифф должен быть пустым для пары «исходник в pythonpkg → файл в otterbrix» с учётом согласованного сопоставления путей).
- Любой непустой дифф по содержимому вне явно описанного исключения (например, только если инструмент переноса исказил EOL и это зафиксировано) означает нарушение границы PR1.

## Ветвление (логически)

- `import/pythonpkg-<short-hash>` (или эквивалентное имя) от целевой базы `otterbrix` (какую линию выбрать для форка — отдельно согласовать с владельцем репозитория).
- После merge PR1: ветка с доработками для PR2 от этой точки.

## Риски и смягчение

- **Ревью PR1** почти исключительно структурное (список файлов, соответствие hash), не функциональное.
- **CI:** для PR1 либо не требовать зелёных проверок, либо отключить/пометить ожидаемые падения; для PR2 — обычные требования качества.
- **Большой дифф:** PR1 может быть объёмным; снижается за счёт отсутствия смешивания с логикой диплома.

## Следующий шаг после утверждения spec

Создать план реализации (отдельный документ в `docs/superpowers/plans/`) с конкретными git-командами, сопоставлением каталогов `pythonpkg` → `otterbrix`, и порядком открытия PR в `prawwtocol/otterbrix`.
